## 문제 목록

### [긴급 — 즉시 수정 필요]

#### 문제 1: 비동기 이벤트 루프를 블로킹하는 동기 SQLite I/O

**위치**: 계획서 §3-2, `log_chat_message()` 핸들러 삽입 코드

계획서의 `log_chat_message()`는 **동기(sync) 함수**인데, `text_message_handler`는 `async` 함수다.  
아래처럼 `await` 없이 직접 호출하면 `sqlite3.connect()` 의 블로킹 파일 I/O가 **asyncio 이벤트 루프 전체를 멈춘다**.

```python
# 계획서 제안 코드 (문제 있음)
async def text_message_handler(update: Update, context: ContextTypes.DEFAULT_TYPE):
    text = update.message.text.strip()
    log_chat_message(sender="user", message_text=text, session_id=update.message.chat_id)  # 이벤트 루프 블락
```

`telegram_bot.py`의 `update_command`(L198)는 이미 `asyncio.to_thread(agent.run_relay)`로 블로킹 작업을 스레드에 위임하고 있다. 동일한 패턴을 적용해야 한다.

**수정 방향**: `log_chat_message()`를 `asyncio.to_thread(log_chat_message, ...)`로 감싸거나, `aiosqlite` 라이브러리로 비동기 SQLite를 사용해야 한다.

#### 문제 2: `message_text NOT NULL` 제약 — 텍스트 없는 메시지에서 INSERT 실패

**위치**: 계획서 §2 스키마 명세, `message_text TEXT NOT NULL`

텔레그램은 사진, 스티커, GIF, 파일 등 **텍스트가 없는 메시지**를 전송할 수 있다. 이 경우 `update.message.text`가 `None`을 반환하는데, `message_text TEXT NOT NULL` 제약에 걸려 INSERT가 예외를 발생시킨다.

`log_chat_message()`에서 `except`로 잡혀 `print`만 출력하고 조용히 넘어가므로, 운영 중에는 원인 파악이 어렵다.

**수정 방향**: 스키마에서 `NOT NULL` 제거 후 `NULL` 허용, 또는 호출부에서 `text or ""` 처리 추가.

### [설계 결함]

#### 문제 3: `init_chat_db()` 호출 위치 미명세

**위치**: 계획서 §3-1, §1 요구사항 2번

계획서는 "봇 기동 시 테이블 자동 생성을 보장"한다고 했으나, `init_chat_db()`를 **어디에서 호출하는지 코드 수준으로 명세하지 않았다**.

`telegram_bot.py`의 `main()` 함수(L388)를 보면 `Application.builder()` 이후 `app.run_polling()`까지 이어지는 기동 흐름이 있다. 이 흐름 어디에 `init_chat_db()`를 삽입할지 명시되어야 한다.

`post_init` 콜백(L400)이 이미 있으므로 여기에 추가하거나 `main()` 진입부에서 단독 호출하는 방식 중 하나를 계획서에 명시해야 한다.

#### 문제 4: `reply_and_log` 전환 시 기존 핸들러 마이그레이션 작업량 누락

**위치**: 계획서 §3-2 봇 송신 기록 코드

계획서는 봇 송신 기록을 위해 `reply_and_log()` 헬퍼를 신설하고 기존 `reply_text` 호출을 교체하도록 제안한다. 그러나 실제 `telegram_bot.py`에서 `reply_text`를 호출하는 곳은 아래와 같이 **9곳 이상**이다:

| 핸들러 | 호출 위치 |
|---|---|
| `start_command` | L146 |
| `bitcoin_command` | L151 |
| `fed_command` | L156 |
| `korea_command` | L161 |
| `log_command` | L183, L185 |
| `update_command` | L191, L204 |
| `button_callback_handler` | L273, L276, L279, L282, L288, L292, L296 |

이 중 `button_callback_handler`는 `query.edit_message_text()`를 사용하므로 `reply_and_log`로 대체 **불가**다. 해당 핸들러의 봇 응답 로깅 방법이 계획서에 누락되어 있다.

계획서가 승인되어 구현에 착수할 경우, 이 마이그레이션 범위를 반드시 포함해야 한다.

#### 문제 5: "원자적(Atomic)" 주장과 실제 코드의 불일치

**위치**: 계획서 §1 요구사항 2번, `log_chat_message()` 주석

계획서와 코드 주석에서 "원자적으로 SQLite 테이블에 적재"라고 명시하고 있으나, 실제 구현에는 다음이 없다:

- SQLite WAL(Write-Ahead Logging) 모드 미설정
- `threading.Lock` 또는 asyncio `Lock` 미사용
- `check_same_thread=False` 미명시

`asyncio.to_thread()`로 에이전트를 실행(L200)하면서 동시에 텔레그램 핸들러도 처리하는 멀티스레드 혼합 환경에서, WAL 모드 없이 동시 write가 발생하면 SQLite가 `SQLITE_BUSY` 예외를 던질 수 있다. 계획서 §4.1의 검증 항목에 "멀티스레드 순차 호출 검증"이 있으나, 이를 통과시키려면 먼저 Lock 또는 WAL 모드를 추가해야 한다.

### [명세 누락]

#### 문제 6: `created_at` 타임존 미명세

**위치**: 계획서 §2 스키마, §3-1 코드

`datetime.now()`는 **로컬 시스템 시간** 기준이다. 서버가 UTC 설정이면 한국 시각과 9시간 차이가 발생한다. 계획서의 수동 검증 예시(§4.2)에 `2026-05-23 17:02:40`과 같이 구체적인 시각이 명시되어 있으므로, KST 기준인지 UTC 기준인지 스키마 명세에 반드시 기재해야 한다.

#### 문제 7: `session_id` 기본값 `"default"` 위험성 미언급

**위치**: 계획서 §3-1, `log_chat_message(session_id: str = "default")`

`session_id` 기본값이 `"default"`로 설정되어 있다. 만약 호출부에서 `session_id`를 누락하면 서로 다른 사용자의 대화가 `"default"`로 합쳐져 조회 시 혼재된다. 계획서에서 이 기본값의 의미 및 오남용 위험을 명시해야 한다.

실제 운용에서는 항상 `chat_id`를 넘기도록 강제하거나, 기본값을 `None`으로 두고 호출부에서 필수 인자로 처리하는 것이 안전하다.

---

#### 문제 8: 로그 누락 시 알림 수단 없음

**위치**: 계획서 §3-1, `log_chat_message()` 예외 처리

```python
except Exception as e:
    print(f"[!] SQLite 대화 로그 저장 실패 (발신자: {sender}): {e}")
```

예외를 `print`로만 처리하므로, 운영 중 SQLite 저장 실패가 반복되어도 사용자나 운영자는 인지할 수 없다. 대화 로그 적재가 핵심 기능이라면 에러 로그를 파일에 기록하거나 텔레그램으로 알림을 보내는 방안을 계획서에 명시해야 한다.
