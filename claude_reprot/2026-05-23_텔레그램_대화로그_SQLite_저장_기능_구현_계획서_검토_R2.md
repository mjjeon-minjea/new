**작성일**: 2026-05-23
**검토 대상**: `antigravity_report/plan/2026-05-23_plan_텔레그램_대화로그_SQLite_저장_기능_구현_계획서_R2.md`
**검토 방법**: 계획서 전문 독해 + `telegram_bot.py` (430줄) 및 `agents/shared/config.py` (52줄) 실물 코드 직접 대조
**수정번호**: R2

## 1. 총평

R2 계획서는 R1 검토에서 제기된 7대 지적사항(비동기 루프 런타임 오류, 슬래시 커맨드 입력 누락, FileHandler 초기화 순서, 상태 메시지 로깅 누락, 브리핑 푸시 로깅 누락, WAL 설계 명세 불완전, config.py 관리체계 누락)에 대해 기술적으로 정확하고 논리적인 해결책을 제시하고 있으며, 전반적인 설계 품질이 상당히 높습니다.

다만 실물 코드(`telegram_bot.py`)와 정밀 대조한 결과, **신규 결함 4건, 설계 개선 권고 3건**이 추가로 발견되었습니다. 이 항목들은 R1 검토 범위를 벗어난 신규 리스크이며, 아래에 우선순위 순으로 명세합니다.

## 2. R1 지적사항 반영 검증

|:---:|:---|:---|:---:|

## 3. 신규 발견 결함 명세

### ① [런타임 버그] `/help` CommandHandler 로깅 완전 누락

**심각도**: 중 (데이터 누락)

실물 `telegram_bot.py`의 `main()` 함수에는 `/help` 명령어가 `start_command`에 별도로 바인딩되어 있습니다.

```python
# telegram_bot.py L411~412 — 실물 코드
app.add_handler(CommandHandler("start", start_command))
app.add_handler(CommandHandler("help", start_command))  # ← 별도 바인딩 존재
```

그러나 R2 계획서 §4.②.① "6대 커맨드 전수 패치" 목록에는 `/help`가 누락되어 있습니다. `/help` 입력 시 `start_command` 핸들러를 타지만, 해당 핸들러에서 `await log_chat_message(sender="user", message_text="/start", ...)` 처럼 하드코딩된 텍스트로 기록하면 실제 사용자 입력(`/help`)이 아닌 잘못된 값이 DB에 저장됩니다.

**권고 조치**: `start_command` 내 로깅 호출을 하드코딩 대신 `update.message.text`로 실제 입력값을 캡처하도록 수정.
```python
async def start_command(update: Update, context: ContextTypes.DEFAULT_TYPE):
    actual_input = update.message.text or "/start"  # /start 또는 /help 모두 정확히 기록
    await log_chat_message(sender="user", message_text=actual_input, session_id=update.message.chat_id)
    ...
```
이 방식은 6대 커맨드 모두에 동일하게 적용하는 것이 권장됩니다.

### ② [설계 충돌] `update_command`의 `status_msg.delete()` vs 로깅 순서 충돌

**심각도**: 중 (설계 일관성)

실물 `telegram_bot.py` L203에서 `status_msg`는 성공 시 `delete()`로 삭제됩니다.

```python
# telegram_bot.py L203~204 — 실물 코드
await status_msg.delete()  # 상태 메시지 삭제
await update.message.reply_text(briefing_message, parse_mode="Markdown")
```

R2 계획서 §4.②.③에서는 이 `status_msg`에 대한 로깅을 최상단에서 수행하도록 명세하고 있으나, **삭제된 메시지를 로깅할지 여부 및 `delete()` 호출 유지 여부**에 대한 명세가 없습니다. 이 충돌로 인해 구현자가 판단해야 하는 모호한 상황이 발생합니다.

**권고 조치**: 계획서에 다음 중 하나를 명확히 결정하여 명세 추가.
- 방안 A: `status_msg` 발송 즉시 로깅 → `delete()` 유지 (로그에는 삭제된 메시지도 기록)
- 방안 B: `delete()` 제거 → `edit_text()`로 대체하여 삭제 없이 메시지 유지 및 로깅 일원화 (UX 변경 수반)

### ③ [런타임 버그] `--send-briefing` 모드 실행 시 `init_chat_db()` 미호출 경로

**심각도**: 높음 (데이터 유실)

실물 코드에서 `--send-briefing` 인자로 직접 실행하면 `main()` 함수를 거치지 않고 `asyncio.run(send_briefing_notification())`를 직접 호출합니다.

```python
# telegram_bot.py L426~429 — 실물 코드
if len(sys.argv) > 1 and sys.argv[1] == "--send-briefing":
    asyncio.run(send_briefing_notification())  # ← main() 우회, init_chat_db() 미호출
else:
    main()
```

R2 계획서는 `init_chat_db()` 호출을 `main()` 내 `is_token_valid()` 직후로만 명세하고 있습니다. 따라서 `--send-briefing` 모드 실행 시 `telegram_chat.db`와 테이블이 미생성된 상태에서 `log_chat_message`가 호출되어 `sqlite3.OperationalError: no such table: telegram_chat_logs`가 발생하고 브리핑 로그가 유실됩니다.

**권고 조치**: `--send-briefing` 분기에도 `init_chat_db()`를 선행 호출하거나, `log_chat_message` 내 `_sync_insert`에서 테이블 부재 시 자동 생성하는 방어 로직 추가.
```python
if len(sys.argv) > 1 and sys.argv[1] == "--send-briefing":
    init_chat_db()  # ← 반드시 추가
    asyncio.run(send_briefing_notification())
```

### ④ [설계 결함] `_send_emergency_notification` 내 이중 임포트 중복

**심각도**: 낮음 (코드 품질)

R2 계획서 §3.①의 `_send_emergency_notification` 함수 내에서 `TELEGRAM_BOT_TOKEN`과 `TELEGRAM_CHAT_ID`를 동적 임포트합니다.

```python
# 계획서 코드 (§3.①)
from agents.shared.config import TELEGRAM_BOT_TOKEN, TELEGRAM_CHAT_ID
```

그러나 실물 `telegram_bot.py` 상단에서 이미 `BOT_TOKEN = TELEGRAM_BOT_TOKEN`과 `CHAT_ID = TELEGRAM_CHAT_ID`로 전역 바인딩이 완료되어 있습니다 (L27~28). 서브 스레드 내 중복 임포트는 기능에는 영향이 없으나 불필요한 모듈 접근을 유발합니다.

**권고 조치**: 동적 임포트 대신 모듈 수준 전역 변수를 직접 참조.
```python
# 개선안
if BOT_TOKEN and CHAT_ID:  # 이미 telegram_bot.py 전역에 바인딩된 변수 재사용
    emergency_bot = Bot(token=BOT_TOKEN)
    ...
```

## 4. 설계 개선 권고

### ① `session_id` 타입 힌트 불일치

`log_chat_message`의 시그니처에서 `session_id: str`로 타입 힌트를 정의하고 있으나, 모든 핸들러 호출부에서 `update.message.chat_id` (Python `int`)를 그대로 전달합니다. `_sync_insert` 내에서 `str(session_id)` 변환이 이루어지므로 런타임 에러는 없지만, 타입 안전성 측면에서 호출부를 `str(update.message.chat_id)`로 명세하거나 시그니처를 `Union[str, int]`로 수정하는 것이 권장됩니다.

### ② `msg.text` None 가드 미비

`await update.message.reply_text(...)` 반환 객체의 `.text` 속성은 Markdown 파싱 실패나 특수 메시지 유형에서 `None`이 될 수 있습니다. 계획서 전반에 걸쳐 `msg.text`를 직접 `log_chat_message`에 전달하는 패턴이 반복되는데, `message_text` 파라미터 자체가 `None` 허용(`NULL`) 설계이므로 `msg.text or ""`로 방어적 처리를 명세에 추가하는 것이 일관성을 높입니다.

### ③ `button_callback_handler`의 `edit_message_text` 반환값 검증

`query.edit_message_text()` 호출 시 반환값을 `msg`로 받아 `msg.text`를 로깅에 사용하는 패턴은, python-telegram-bot v20에서 `edit_message_text()`가 수정된 `Message` 객체를 반환하므로 기술적으로 올바릅니다. 다만 이미 삭제된 메시지에 `edit_message_text()`를 시도할 경우 `BadRequest` 예외가 발생하므로, 해당 예외에 대한 로깅 처리도 명세에 포함하는 것이 견고성을 높입니다.

## 5. 실물 코드 현황 확인

R2 계획서는 신규 구현 명세로서 사용자 승인 후 실물 파일에 패치될 예정이며, 위 검토 결과를 반영하여 R3 보완 또는 구현 단계에서 직접 조치를 권고합니다.
