## ⚠️ 안티그래비티의 반복 패턴 경고 (R0 → R1 → R2 → R3 공통 지적)

본 검토에서 가장 먼저 짚어야 할 것은 기술 결함보다 **4회 연속으로 반복되는 설계 습관**입니다. 이 패턴이 교정되지 않으면 R4, R5로 검토가 계속 연장됩니다.

---

### 반복 패턴 1: "완전무결" 자기선언과 실제 완성도의 반복적 괴리

R1, R2, R3 계획서 모두 서두에 동일한 구조의 문장이 등장합니다.

- R1: *"완벽하게 보완 및 반영하여 봇 기동 시 단 하나의 에러나 누락도 없이 완전무결한 프로덕션 기동을 담보"*
- R2: *"실물 파일 패치 시 컴파일/런타임 실패 리스크를 제로(0)화"*
- R3: *"남김없이 완벽하게 적용한 최고 신뢰도의 설계도"*

그러나 매 버전마다 신규 런타임 버그와 설계 결함이 발견되었습니다. **자기검증 없이 완성도를 선언하는 관행 자체가 리스크입니다.** 이후 계획서에서는 이러한 수사를 배제하고, 스스로 검토한 미결 리스크를 마지막 섹션에 솔직하게 명시하는 방식으로 전환하기를 강력히 권고합니다.

---

### 반복 패턴 2: 복잡한 핸들러는 추상적 명세로 처리하고 구체적 코드 스니펫 생략

R1부터 R3까지, `bitcoin_command`, `fed_command`, `korea_command`, `log_command`, `send_briefing_notification` 이 5개 핸들러는 항상 "이렇게 하면 됩니다" 수준의 추상적 명세만 반복됩니다. 반면 `start_command`와 `update_command`는 매번 상세한 코드 스니펫이 제공됩니다.

이것은 구조적 편향입니다. **명세서에서 코드 스니펫이 없는 곳이 바로 구현 시 판단 오류가 발생하는 곳**입니다. 아래 §3.②에서 이번 R3에서도 같은 이유로 발생한 미완성 명세를 기술합니다.

---

### 반복 패턴 3: 실물 코드의 기존 구조를 직접 대조하지 않는 관행

계획서는 매번 "이 코드를 추가하면 됩니다" 방식으로 작성되는데, 실물 `telegram_bot.py`의 **기존 try-except 구조, 다중 분기, 반환값 처리 패턴**과 맞닿는 지점에서 충돌이 반복적으로 발생합니다. 계획서를 작성하기 전에 실물 코드의 해당 함수 전체를 인용하고, 그 위에 패치를 명세하는 방식으로 전환하면 이 패턴을 끊을 수 있습니다.

## 3. 신규 발견 결함 명세

### ① [설계 충돌] `button_callback_handler`의 신규 try-except와 기존 구조 충돌

**심각도**: 중

실물 `button_callback_handler` (L233~296)는 이미 내부에 `approve`/`reject` 분기 각각에 `try-except Exception as e` 블록이 존재합니다. R3 계획서 §4.②.④가 제시하는 코드는 이 기존 구조를 무시하고 외부에 새로운 try-except를 덧씌우는 방식입니다.

```python
# R3 계획서 §4.②.④
try:
    msg = await query.edit_message_text(text=..., parse_mode="Markdown")
    await log_chat_message(...)
except Exception as callback_err:
    ...
```

문제는 두 가지입니다. 첫째, 이 스니펫이 기존 `approve`/`reject` 분기 각각의 어느 위치에 삽입되어야 하는지 명세가 없습니다. 둘째, 기존 내부 `try-except`가 예외를 먼저 잡으면 새 `try-except`에는 예외가 전달되지 않으므로, 두 구조가 중첩될 경우 어느 쪽이 우선하는지 정의가 필요합니다.

**권고 조치**: 실물 코드의 `approve` 분기 전체를 계획서에 인용한 뒤, 로깅 삽입 위치와 기존 try-except와의 관계를 명시적으로 명세.

---

### ② [명세 불완전] `bitcoin_command` / `fed_command` / `korea_command` / `log_command` 봇 응답 로깅 코드 미제공

**심각도**: 중 (반복 패턴 2의 직접적 재발)

R3 §4.②.②의 명세는 다음과 같습니다:

> *"송출 코드 직후 `msg = await update.message.reply_text(...)` 반환값을 비동기 저장합니다."*

이것은 R1, R2에서도 동일하게 반복된 추상적 명세입니다. 실물 코드와 대조하면 세 가지 문제가 드러납니다.

**문제 A**: `bitcoin_command`, `fed_command`, `korea_command` 실물 코드는 반환값을 받지 않고 있습니다.
```python
# 실물 코드 (L151, L156, L161)
await update.message.reply_text(f"🪙 *[Bitcoin]...*")  # msg = 없이 await만
```
`msg = await ...` 형태로 변경해야 하는데, 이것이 명세에 없습니다.

**문제 B**: `log_command`는 `reply_text` 호출이 두 곳에 있으며 기존 try-except 구조가 있습니다.
```python
# 실물 코드 (L167, L183, L185)
if not log_file.exists():
    await update.message.reply_text("📂 에이전트 활동 로그가 아직 없습니다.")  # 첫 번째
    return
try:
    ...
    await update.message.reply_text(log_text)   # 두 번째
except Exception as e:
    await update.message.reply_text(f"❌ 로그 조회 오류: {e}")  # 세 번째
```
세 곳 모두 로깅이 필요한데, 어디에 어떻게 삽입해야 하는지 전혀 명세되지 않았습니다.

**권고 조치**: 이 4개 핸들러 각각에 대해 실물 코드 전체를 인용하고, 패치 후 전체 코드를 스니펫으로 명세.

---

### ③ [명세 불완전] `send_briefing_notification` 로깅 위치와 기존 try-except 통합 방식 미정의

**심각도**: 중

실물 `send_briefing_notification` (L330~334)는 try-except로 `send_message()`를 감싸고 있습니다.

```python
# 실물 코드 (L330~334)
try:
    await app.bot.send_message(chat_id=CHAT_ID, text=briefing_message, parse_mode="Markdown")
    print("[+] 텔레그램 일일 브리핑 메시지 발송 완료!")
except Exception as e:
    print(f"[!] 텔레그램 브리핑 발송 실패: {e}")
```

R3 §4.②.⑤의 명세는 try-except 없이 두 줄을 순차 나열합니다.

```python
# R3 계획서 명세
await app.bot.send_message(...)
await log_chat_message(...)
```

이 명세대로 기존 try-except 안에 `log_chat_message`를 넣으면, `send_message()`가 예외로 실패할 경우 로깅도 함께 skip됩니다. `send_message()` 성패와 무관하게 항상 로깅하려면 try 외부에 위치해야 합니다. 이 설계 결정이 명세에 없습니다.

**권고 조치**: 기존 try-except 구조를 계획서에 명시하고, 로깅을 try 블록 내부(발송 성공 시에만 로깅)에 둘지, try 블록 외부(항상 로깅)에 둘지 명시적으로 결정 및 이유 기술.
