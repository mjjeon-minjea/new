## 1. 총평

R4는 이 시리즈 전체를 통틀어 가장 큰 질적 도약을 이뤄냈습니다.

Before/After 코드 스니펫이 모든 핸들러에 빠짐없이 제공되었고, Known Unknowns 섹션이 신설되었으며, `button_callback_handler`의 구조 충돌과 `send_briefing_notification`의 try-except 통합 방식이 모두 명확하게 결정되었습니다. 반복 패턴 지적의 효과가 이번 버전에서 처음으로 온전히 나타났습니다.

신규 발견 결함은 **1건**입니다. 중요한 점은, 이번 결함이 새로운 부주의가 아니라 **계획서 자체가 문제를 인지하고 있음에도 코드로 해결하지 못한 경우**라는 점입니다.

## 3. 신규 발견 결함

### ① [런타임 버그] `text_message_handler`와 커맨드 함수 간 사용자 입력 이중 로깅

**심각도**: 중 (데이터 오염 — DB에 동일 발화가 2줄 기록)

R4 §4.⑦의 After 코드에서 `text_message_handler`는 진입 직후 사용자 입력을 먼저 로깅합니다.

```python
# text_message_handler After (§4.⑦)
text = update.message.text.strip() if update.message.text else ""
await log_chat_message(sender="user", message_text=text, ...)  # ← 1회 로깅

if any(kw in text for kw in [...]):
    await update_command(update, context)  # ← 내부에서 또 로깅
elif "비트코인" in text or ...:
    await bitcoin_command(update, context)  # ← 내부에서 또 로깅
```

그런데 R4 §4.②에서 `bitcoin_command` 등 각 커맨드 함수도 내부 최상단에서 동일하게 사용자 입력을 로깅합니다.

```python
# bitcoin_command After (§4.②)
actual_input = update.message.text or "/bitcoin"
await log_chat_message(sender="user", message_text=actual_input, ...)  # ← 2회 로깅
```

결과적으로 사용자가 "비트코인"이라고 입력하면 DB에는 동일 내용이 두 줄 삽입됩니다. 타임스탬프도 거의 동일하여 실제 로그 분석 시 혼란을 유발합니다.

**계획서 자체도 이 문제를 인지하고 있습니다.** §4.⑦ After 코드 주석에 다음과 같이 명시되어 있습니다.

> *"각 커맨드 내부에서 사용자 로깅이 중복으로 찍히지 않도록, text_message_handler 내부 분기 진입 시에는 커맨드 함수들이 직접 처리하도록 합니다."*

그러나 실제 After 코드는 이 주석 의도와 반대로 동작합니다. `text_message_handler`에서 먼저 로깅한 뒤 커맨드 함수를 호출하여 내부에서 다시 로깅하는 구조입니다.

**권고 조치 (2가지 중 선택):**

방안 A — `text_message_handler`에서 로깅 제거 (각 커맨드 함수에서만 로깅):
```python
async def text_message_handler(update, context):
    text = update.message.text.strip() if update.message.text else ""
    # 로깅 없음 — 분기된 커맨드 함수 내부에서 처리
    if any(kw in text for kw in [...]):
        await update_command(update, context)
    ...
```

방안 B — 각 커맨드 함수에서 로깅 제거하고 `text_message_handler`에서만 처리 (단, 슬래시 직접 호출 시 로깅 누락 발생하므로 권장하지 않음):

**방안 A가 권장됩니다.** 슬래시 커맨드 직접 호출(`/bitcoin`)과 자연어 입력("비트코인") 모두 각자의 경로에서 로깅되므로 누락 없이 일원화됩니다.

## 4. 경미한 잔존 항목

### Known Unknowns 품질에 대한 코멘트

Known Unknowns 1번 항목은 "WAL+Lock으로 예방했으나 하드웨어 성능 저하 환경에서의 완전 차단은 편차가 있을 수 있다"는 내용으로, 이미 대응한 리스크에 대한 설명에 가깝습니다. Known Unknowns 섹션의 본래 취지는 **아직 해결하지 못한 것**을 솔직하게 기재하는 것입니다. 2번 항목(미디어 메시지 메타데이터 유실)이 훨씬 적절한 형태입니다. 다음 계획서부터는 이미 대응한 리스크는 별도로, 미해결 불확실성만 Known Unknowns에 기재하도록 구분하는 것이 좋습니다.
