#### 문제 1: `_send_emergency_notification` 내 `asyncio.get_event_loop()` — 스레드 내 RuntimeError

**위치**: 계획서 §3-1, `_send_emergency_notification` 함수

계획서의 비상 알림 코드는 다음 구조다:

```python
def _sync_insert():      # asyncio.to_thread()로 실행되는 동기 함수
    with db_lock:
        try:
            ...
        except Exception as e:
            _send_emergency_notification(str(e))

def _send_emergency_notification(err_msg: str):
    ...
    loop = asyncio.get_event_loop()   # ← 여기가 문제
    if loop.is_running():
        loop.create_task(emergency_bot.send_message(...))
```

`_sync_insert()`는 `asyncio.to_thread()`로 **메인 이벤트 루프와 다른 스레드 풀**에서 실행된다. 이 스레드에는 실행 중인 이벤트 루프가 없으므로:

- **Python 3.10+**: `asyncio.get_event_loop()`가 `DeprecationWarning`을 발생시키고 새 루프를 생성한다. 이 새 루프는 실행 중이 아니므로 `loop.is_running()`이 `False`가 되어 비상 알림이 발송되지 않는다.
- **Python 3.12+**: `asyncio.get_event_loop()`가 실행 중인 루프 없이 호출되면 `RuntimeError`가 발생한다.

결과적으로 R1이 강조하는 "운영자 텔레그램 에러 푸시 메커니즘"이 실제 에러 발생 시 작동하지 않는다.

**수정 방향**: `log_chat_message` (async 함수) 내에서 `asyncio.get_running_loop()`로 루프를 캡처하여 `_sync_insert`에 클로저로 전달하고, 스레드 내에서는 `asyncio.run_coroutine_threadsafe(coro, loop)`를 사용해야 한다.

```python
async def log_chat_message(...):
    loop = asyncio.get_running_loop()   # async 컨텍스트에서 안전하게 캡처
    
    def _send_emergency_notification(err_msg: str):
        ...
        asyncio.run_coroutine_threadsafe(
            emergency_bot.send_message(...), loop   # 올바른 루프 참조
        )
    
    def _sync_insert():
        ...
    
    await asyncio.to_thread(_sync_insert)
```

#### 문제 2: 슬래시 커맨드(Command) 사용자 입력 로깅 완전 누락

**위치**: 계획서 §4-2 ① 사용자 발신 기록 지점

R1은 사용자 입력 로깅을 `text_message_handler` 최상단에서만 처리한다:

```python
async def text_message_handler(update: Update, context: ContextTypes.DEFAULT_TYPE):
    text = update.message.text.strip() if update.message.text else ""
    await log_chat_message(sender="user", ...)
```

그러나 실제 `telegram_bot.py`의 핸들러 등록을 보면 (L419):

```python
app.add_handler(MessageHandler(filters.TEXT & ~filters.COMMAND, text_message_handler))
```

`~filters.COMMAND` 필터로 인해 `/bitcoin`, `/fed`, `/korea`, `/start`, `/help`, `/update`, `/log` 등 **슬래시 커맨드는 `text_message_handler`를 거치지 않는다**. 각 `CommandHandler`로 직접 라우팅된다.

따라서 사용자가 `/bitcoin`을 입력했을 때 `bitcoin_command`로 직접 이동하여 사용자 입력이 DB에 기록되지 않는다. 계획서 §4-2에서 열거한 마이그레이션 9개소가 모두 **봇 송신(Output)** 위치만 다루고 있어, 커맨드 기반 사용자 입력(Input) 로깅이 통째로 빠진 구조다.

**수정 방향**: 각 커맨드 핸들러 진입부에도 사용자 입력 로깅을 추가해야 한다.

```python
async def bitcoin_command(update: Update, context: ContextTypes.DEFAULT_TYPE):
    await log_chat_message(sender="user", message_text="/bitcoin", session_id=update.message.chat_id)
    report = parse_wiki_for_telegram("Bitcoin")
    ...
```

### [설계 결함]

#### 문제 3: 모듈 레벨 `logging.FileHandler` 초기화 — import 시 파일 시스템 오류로 봇 시작 불가

**위치**: 계획서 §3-1, 전역 객체 초기화 블록

```python
chat_log_dir = ROOT_DIR / "_company" / "logs"
chat_log_dir.mkdir(parents=True, exist_ok=True)
file_handler = logging.FileHandler(chat_log_dir / "chat_error.log", encoding="utf-8")
```

이 코드는 `telegram_bot.py` 모듈 상단에 위치하므로, `import telegram_bot` 또는 `python telegram_bot.py` 실행 순간에 파일 시스템에 접근한다. 디스크 권한 문제, 경로 이상, 또는 `ROOT_DIR`이 예상과 다른 디렉토리를 가리킬 경우 `FileHandler` 생성이 실패하여 **봇 자체가 시작되지 않는다**.

기존 `config.py`는 디렉토리 생성을 모듈 레벨에서 수행하지만, 파일 핸들러 생성까지 모듈 레벨에서 하는 것은 과도하다.

**수정 방향**: `logging` 핸들러 초기화를 `init_chat_db()` 내부로 이동시켜 명시적 호출 시점에만 실행되도록 한다.

#### 문제 4: `update_command` 초기 상태 메시지 및 에러 메시지 로깅 누락

**위치**: 계획서 §4-2, `update_command` (L191, L204) 마이그레이션 명세

R1이 명세한 `update_command` 로깅 대상:
- 중복 가동 대기 안내 메시지 (L191)
- 릴레이 완료 브리핑 메시지 (L204)
- 승인 대기 카드 메시지

누락된 대상:
- **초기 상태 메시지** (`status_msg`, L194): 가동 시작 시 보내는 "🔄 *1단계: 0선 금융 데이터를 선행 수집 중...*" 메시지
- **에러 발생 시 편집 메시지** (L229): `status_msg.edit_text(f"❌ 에이전트 릴레이 가동 중 오류...")`

두 경우 모두 봇이 실제로 송신하는 메시지임에도 로깅 명세에서 누락되어, 완전한 대화 기록이 불가능하다.

#### 문제 5: `send_briefing_notification()`의 자동 브리핑 메시지 로깅 완전 누락

**위치**: 계획서 §4 전체, `scheduled_update_loop` / `send_briefing_notification`

12시간 자동 스케줄러(`scheduled_update_loop`, L365)가 호출하는 `send_briefing_notification()`(L300)은 `app.bot.send_message(chat_id=CHAT_ID, text=briefing_message)`로 텔레그램 채널에 자동 브리핑을 푸시한다(L331).

이 메시지는 R1 계획서 전체에서 로깅 대상으로 단 한 번도 언급되지 않는다. "봇이 송신하는 모든 메시지를 기록"한다는 목표와 정면 배치된다.

### [명세 불완전]

#### 문제 6: `threading.Lock` + WAL 병용의 설계 의도 미명시

**위치**: 계획서 §3-1, `db_lock` 및 `init_chat_db` WAL 설정

WAL(Write-Ahead Logging) 모드의 목적은 **동시 다중 reader + 단일 writer**를 허용하는 것이다. 그런데 R1은 WAL과 함께 `threading.Lock`을 사용해 write를 완전히 순차화한다.

이 시스템의 실제 사용 패턴(단일 봇 프로세스, 대화 빈도 낮음)에서 두 메커니즘을 병용할 경우:
- **Lock**: write 직렬화 보장 (충분)
- **WAL**: 동시 reader를 위한 오버헤드 추가 (이 시스템에서 실질 이득 없음)

"WAL + Lock 병용"이 보수적 안전망인지, 아니면 향후 read 쿼리 동시성을 고려한 의도적 설계인지 계획서에 명시되어야 한다. 현재대로라면 WAL의 이점 없이 오버헤드만 추가되는 구성이다.

#### 문제 7: `_company/logs/` 디렉토리가 `config.py` 관리 체계에서 누락

**위치**: 계획서 §3-1, 로거 초기화 코드 vs. `agents/shared/config.py`

`config.py`는 `_company/` 하위 디렉토리를 중앙에서 관리하며 자동 생성한다(L29~37):

```python
APPROVALS_PENDING_DIR = ROOT_DIR / "_company" / "approvals" / "pending"
APPROVALS_APPROVED_DIR = ROOT_DIR / "_company" / "approvals" / "approved"
SESSIONS_DIR = ROOT_DIR / "_company" / "sessions"
```

R1에서 신설하는 `_company/logs/`는 이 체계에 등록되지 않고 `telegram_bot.py` 모듈 레벨 코드에서 독자적으로 생성한다. 경로 상수를 `config.py`에 추가(`CHAT_LOG_DIR` 등)하고 중앙에서 관리하는 방향으로 명세를 보완해야 한다.
